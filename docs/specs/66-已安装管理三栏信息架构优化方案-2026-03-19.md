# 已安装管理三栏信息架构优化方案（仅分析，待确认后实施）

日期：2026-03-19
状态：方案评审
范围：仅针对“已安装管理”页面交互结构，不涉及 OpenClaw 专区与其他页面

## 1. 现状问题（基于代码与界面）

### 1.1 当前三栏语义冲突
当前结构：
- 左栏：工具筛选（HostFilterItem）
- 中栏：仍是工具概览（HostOverviewRow）
- 右栏：工具详情或组件详情

代码证据：
- 左栏/中栏都绑定同一类 host 集合：`visibleHostEntries`
  - `src/components/pages/installed-management.tsx` 中 `leftColumn` 与 `middleColumn` 都在遍历 host。
- 主状态是 `selectedHostId` + `selectedItem`，但中栏没有承担“第 2 级选择”的职责。

结果：
- 第 1 步和第 2 步都是“选工具”，上下关系断裂。
- 用户会以为中栏是“同级新列表”，不是“当前工具的子项”。

### 1.2 零基础用户认知负担高
- 没有“第 1 步/第 2 步/第 3 步”显式引导。
- 右栏在“未选组件”时承载过多内容（风险、统计、依据、组件列表），导致焦点不清。
- 操作入口（批量修复/更新/清理）与当前选择上下文关系不够显性。

## 2. 官方最佳实践依据（研究结论）

### 2.1 Apple（分栏导航）
核心点：分栏应是层级关系：sidebar -> content -> detail，detail 由前一列选择驱动；无选择时显示占位状态。
来源：
- https://developer.apple.com/videos/play/wwdc2022/10058/
  - 明确提到 2/3 列 split view；detail 由 sidebar 链接驱动；无选择显示 placeholder。

### 2.2 Android（List-Detail）
核心点：大屏下 list 与 detail 并排；在 list 选中后更新 detail；小屏可用回退在 list/detail 间切换。
来源：
- https://developer.android.com/develop/ui/views/layout/responsive-adaptive-design-with-views
  - `SlidingPaneLayout for list-detail UIs`：选中 list 项后展示 detail，保持上下关系。

### 2.3 WCAG（可预测导航）
核心点：重复导航机制应保持一致顺序，避免让用户每列都重新理解规则。
来源：
- https://www.w3.org/WAI/WCAG22/Understanding/consistent-navigation.html

## 3. 最佳结构结论（推荐）

## 推荐 IA：严格三步链路（1->2->3）
1. 左栏（Step 1）：选择工具（Host）
2. 中栏（Step 2）：仅显示“该工具下的组件列表”（MCP/Skill/插件）
3. 右栏（Step 3）：显示“当前组件详情 + 操作按钮”

当未选组件时：
- 右栏显示“该工具概览（轻量）+ 请在中栏选择组件”的占位提示。
- 不再把完整组件列表放在右栏，避免和中栏重复。

这套结构最符合你说的：
- 先选第一个界面（工具）
- 再选第二个界面（该工具下组件）
- 再在第三个界面执行动作（修复/更新/卸载/信任策略）

## 4. 具体布局建议（不改视觉语言，只改信息关系）

### 4.1 左栏（工具）
保留现有：
- “只看需要处理的工具”开关
- 工具列表 + 风险/能力标签

新增/调整：
- 顶部加步骤文案：`1 选择工具`
- 当前选中工具高亮更强（边框+背景）

### 4.2 中栏（组件列表）
替换当前“工具重复列表”为：
- 标题：`2 选择组件`
- 仅渲染 `items.filter(item => item.platform_id === selectedHost.id)`
- 每行显示：组件名、版本、安全级别、类型（MCP/Skill）
- 默认自动选中第一条高风险组件（可选）

### 4.3 右栏（详情与动作）
- 标题：`3 查看详情并处理`
- 只展示当前组件详情（InstalledItemDetail）
- 当 `selectedItem == null` 时，展示工具概览卡 + 明确 CTA：
  - `请在中栏选择组件`

### 4.4 底部操作条
- 如果已选组件：主按钮优先组件级动作（更新/卸载/修复）
- 如果只选工具未选组件：主按钮优先工具级批量动作（批量修复/批量更新）
- 如果未选工具：底部按钮禁用并提示先完成 Step 1

## 5. 交互状态机（建议）

状态：
- `S0`: 未选工具
- `S1`: 已选工具，未选组件
- `S2`: 已选组件

转换：
- `S0 -> S1`: 点击左栏工具
- `S1 -> S2`: 点击中栏组件
- `S2 -> S1`: 右栏“返回工具概览”
- 切换左栏工具：任何状态回到新工具的 `S1`

## 6. 为什么这套最适合零基础用户

1. 认知路径单线化（工具 -> 组件 -> 操作），减少迷路。
2. 每列职责唯一，避免“左中都选工具”的冲突。
3. 选择关系可见（上一步决定下一步内容），符合“可预测导航”。
4. 降低误操作：没有选到组件时，不展示误导性强的组件级按钮。

## 7. 最小改动实施路径（你确认后执行）

1. 不改 `ThreeColumnLayout` 组件，仅改 `installed-management.tsx` 的列内容映射。
2. 中栏改为组件列表（复用 `InstalledItemRow`）。
3. 右栏从 `HostOverviewDetail` 中移除“组件列表块”，保留风险概览与提示。
4. 增加步骤标题文案（1/2/3）。
5. 跑 `typecheck` + 页面 smoke 测试。

## 8. 验收标准（实施后）

1. 左栏只做工具选择。
2. 中栏只显示当前工具的组件。
3. 右栏只显示当前组件详情（或工具概览占位）。
4. 用户首次进入页面，不看说明也能完成一次“选工具->选组件->执行操作”。
