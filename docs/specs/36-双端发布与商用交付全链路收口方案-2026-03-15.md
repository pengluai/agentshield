# 36. 双端发布与商用交付全链路收口方案

- 日期：2026-03-15
- 适用项目：AgentShield
- 状态：执行中
- 作者：Codex

## 1. 目标

本次收口要解决 4 个直接影响对外发布的问题：

1. GitHub Release 页面当前只显示 macOS 资产，Windows 用户无法直接下载。
2. 当前下载页文案已经完善，但还没有达到“拿来就发”的双端交付状态。
3. 激活码底层签发能力已具备，但缺少面向运营的稳定发码入口与操作文档。
4. 自动支付后自动发码仍依赖真实平台配置，需要明确哪些我可以直接完成，哪些必须由外部账号完成。

## 2. 研究结论

### 2.1 Tauri / GitHub Release 最佳实践

根据 Tauri 官方 GitHub 发布流水线文档，`tauri-action` 的推荐写法是显式配置：

- `tagName`
- `releaseName`
- `releaseBody`
- `releaseDraft`
- `prerelease`

官方示例明确展示了这些字段，而不是依赖隐式默认行为。  
来源：

- https://v2.tauri.app/distribute/pipelines/github/
- https://github.com/tauri-apps/tauri-docs/blob/v2/src/content/docs/distribute/Pipelines/github.mdx

### 2.2 GitHub Actions Artifact 与 Release Asset 是两套东西

GitHub 官方文档说明，workflow artifact 只是“工作流之间共享与保存构建产物”的机制，不等于 release asset。  
这意味着“Windows 构建成功并产出 artifact”不代表“Windows 安装包一定会出现在 Release 页面”。

来源：

- https://docs.github.com/en/actions/tutorials/store-and-share-data

### 2.3 Lemon Squeezy 自动发码链路的最低要求

Lemon 官方文档确认：

1. Webhook 需要真实 `Callback URL`
2. 每个请求都要用签名 secret 做 HMAC 校验
3. 最低建议订阅 `order_created`
4. 若启用订阅，还需要 `subscription_created`、`subscription_payment_success`、`subscription_updated`
5. 若启用内建 license，还会有 `license_key_created`

来源：

- https://docs.lemonsqueezy.com/help/webhooks
- https://docs.lemonsqueezy.com/help/webhooks/signing-requests
- https://docs.lemonsqueezy.com/help/webhooks/event-types

### 2.4 Node 官方对 webhook 验签的要求

Node 官方文档说明，敏感摘要比较应使用 `crypto.timingSafeEqual`，且周边逻辑也要避免引入时序差异。  
这与 Lemon 官方 Node 验签示例一致。

来源：

- https://nodejs.org/api/crypto.html#cryptotimingsafeequala-b

## 3. 基于当前代码与仓库状态的判断

### 3.1 为什么当前 Release 只有 macOS 资产

当前高概率根因不是“Windows 没构建”，而是“Windows 构建产物没有正确附着到 Release 页面”。

已知现象：

1. 发布 workflow 包含 `windows-2022` matrix。
2. workflow artifact 中存在 Windows 安装包。
3. 当前 draft release 页面只出现 macOS 资产。
4. `publish-pilot-artifacts.yml` 使用 `tauri-action`，但没有显式给出 `tagName/releaseName/releaseBody`。
5. 对照仓库内另一个 signed release workflow，可见完整发布参数是显式配置的。

因此，本次优先采取的修复方案是：

1. 把 pilot workflow 改成与 Tauri 官方范式一致的显式 release 配置。
2. 若已有 Windows 构建 artifact，则补做一次 release asset 上传，确保当前版本页面立刻可下载。

### 3.2 为什么现在还不能宣称“自动支付购买后立刻拿码”已经彻底完成

因为这条链路仍取决于两个真实外部配置：

1. 真实 checkout URL
2. 稳定公网 HTTPS 的 license gateway（不能是临时 tunnel）

如果这两个仍是占位值或临时地址，那么代码虽然已具备防呆和 gateway 实现，但不能宣称已经完成正式商用自动化闭环。

### 3.3 激活码其实已经可以发

当前项目已经有本地签发工具链：

1. `issue_activation_code` 二进制可执行
2. `.env.public-sale.local` 中已有签名 seed
3. 当前 public key 与该 seed 已匹配

因此，本次可以把“手动卖码并手动发码”先彻底打通，作为立刻可商用的最小闭环。

## 4. 实施方案

### 4.1 立刻完成的部分

1. 修复 pilot release workflow，使 Windows 资产进入 GitHub Release 页面。
2. 如果当前 draft release 仍缺 Windows 资产，则补传已有安装包。
3. 新增运营侧激活码签发脚本或命令封装，避免手工拼参数。
4. 新建本机交付文档，写清楚：
   - 下载给用户的链接
   - Windows / macOS 分别下载什么
   - 你自己如何生成激活码
   - 激活码如何发送给客户
   - 哪些配置完成后才能切到自动支付发码

### 4.2 条件满足后继续完成的部分

1. 接入真实 checkout URL
2. 部署稳定公网 HTTPS gateway
3. 配置 Lemon webhook
4. 做一单真实或测试支付验收

## 5. 本轮修改清单

预计本轮将修改或新增以下内容：

1. `.github/workflows/publish-pilot-artifacts.yml`
2. 许可证/发码相关辅助脚本
3. 交付与运营文档
4. 必要的测试或校验脚本

## 6. 验收标准

### 6.1 发布验收

1. GitHub Release 页面能同时看到 macOS 与 Windows 可下载资产。
2. Windows 用户至少有一个可直接安装的 `.exe` 或等价安装包。
3. Release 文案说明试用、激活、适用人群、边界。

### 6.2 激活码验收

1. 可以通过单条命令生成一枚有效激活码。
2. 应用内能录入并完成激活。
3. 本机文档中写清楚发码操作与注意事项。

### 6.3 风险验收

1. 不引入新的 placeholder 支付链接。
2. 不允许 public 发布继续使用临时 tunnel gateway。
3. 不在仓库或文档中泄露 secret 原文。

## 7. 假设与边界

### 假设

1. 当前 GitHub CLI 仍已登录且对仓库有写权限。
2. 当前 draft release 允许继续编辑与补充资产。
3. 当前本地环境可以重新触发 GitHub Actions。

### 边界

1. 若没有真实支付平台配置，本轮最多做到“可下载 + 可手动发码 + 可手动卖”。
2. 若没有真实公网服务，本轮不能对外承诺“支付后自动秒发码”。
