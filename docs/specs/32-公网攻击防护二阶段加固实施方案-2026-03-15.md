# 32-公网攻击防护二阶段加固实施方案（2026-03-15）

## 1. Executive Summary
- 目标：在不改变 AgentShield 主交互流程的前提下，降低“公网攻击/提示注入诱导本地宿主执行高危动作”的成功率与影响面。
- 范围：仅改造 `runtime_guard` 运行时守卫，聚焦“检测时效 + 规则覆盖 + 可验证测试”。
- 结果定义：改造完成后，必须通过全量质量门禁（前端、Rust、E2E、发布门禁）。

## 2. Problem Statement and Scope
- 已有能力：AgentShield 已具备运行时高危动作识别、审批、进程暂停与联网拦截链路。
- 当前缺口：
  - 检测时效依赖固定轮询间隔，运行中高风险会话的响应时效可继续提升。
  - 文件删除信号规则对跨平台命令（尤其 Windows/PowerShell）覆盖不完整。
- 本次不做：
  - 不新增公网网关/WAF/反向代理模块。
  - 不重构 UI 或审批流程。
  - 不改动许可证支付链路。

## 3. Current State and Constraints
- 技术栈约束：Tauri + Rust 后端守卫 + React 前端。
- 运行时约束：守卫使用轮询模式识别进程与网络连接，误杀成本较高，需保持“增强检测但可控误报”。
- 交付约束：变更需与现有 `publish-pilot-artifacts` 流程兼容。

## 4. Target Architecture Overview
- 方案 A（本次执行）：
  - 引入“运行中会话加速轮询”策略：有运行会话时缩短守卫轮询间隔；无会话保持常规间隔。
  - 扩展高危文件删除规则：补齐 Windows/PowerShell 常见危险删除指令模式。
  - 配套单测覆盖新增逻辑，并执行全量质量门禁。
- 方案 B（后续候选）：
  - 增加规则热更新与策略灰度（按平台分级 rollout）。
  - 引入高危命令白名单/项目级豁免配置（降低误报）。

## 5. Detailed Component Design
### 5.1 Runtime Poll Interval
- 新增纯函数：根据“策略轮询间隔 + 是否存在 running session”计算有效轮询间隔。
- 设计原则：
  - 有活动会话：优先快速检测（更短间隔）。
  - 无活动会话：保持当前策略值，避免空转资源消耗。

### 5.2 High-Risk File Delete Signals
- 在 `commandline_signals_high_risk_file_delete` 增加跨平台模式：
  - `del /f`, `del /q`, `erase`, `remove-item`, `rd /s /q` 等。
- 保持原有 Linux/macOS 模式，避免回归。

## 6. Data Model and Interface Contracts
- 不新增持久化结构。
- 不新增 Tauri command 接口。
- 仅内部行为变化：`runtime_guard` 轮询调度与规则匹配。

## 7. Non-Functional Requirements
- Security：
  - 对已识别会话更快触发高危检测。
  - 提升 Windows 场景危险删除识别率。
- Reliability：
  - 不破坏现有审批票据链路。
- Performance：
  - 无活动会话不增加额外开销。
- Cost：
  - 仅代码小改，无新增基础设施成本。

## 8. ADRs
### ADR-32-01：优先做“检测时效 + 规则覆盖”而非大规模架构重构
- 决策：执行小步增强，依赖现有守卫架构。
- 备选：
  - 全面事件驱动重构（成本高、风险高）。
  - 保持不变（安全收益不足）。
- 理由：在当前交付节奏下，小步增强具备最高 ROI。

### ADR-32-02：坚持“实现即测试”，并以全量门禁作为发布前置
- 决策：新增单测后必须跑完整质量门禁。
- 理由：安全规则改动容易引入隐性回归，必须强制端到端验证。

## 9. Risk Register
- 风险 1：误报提升导致审批频率上升。`P=3, I=3, Score=9`
  - 缓解：规则保持关键词级最小增量；通过测试样例校验边界。
- 风险 2：轮询加速导致资源占用上升。`P=2, I=3, Score=6`
  - 缓解：仅在 running session 存在时加速，无会话回落常规间隔。
- 风险 3：平台命令兼容性偏差。`P=2, I=4, Score=8`
  - 缓解：加入跨平台命令模式单测并执行发布门禁。

## 10. Delivery Roadmap and Milestones
1. 研究与方案确认（本文件）。
2. 代码改造（轮询策略 + 文件删除规则）。
3. 单测补齐。
4. 全量质量门禁执行。
5. 推送并触发 `publish-pilot-artifacts`。

## 11. Runbook and Observability Baseline
- 本地验证命令：
  - `pnpm lint`
  - `pnpm typecheck`
  - `pnpm test`
  - `pnpm test:e2e`
  - `cargo test --manifest-path src-tauri/Cargo.toml`
  - `pnpm release:github:ready`
- 判定标准：上述全部成功后，才允许触发部署。

## 12. Acceptance Criteria
- 代码层：
  - 存在“运行中会话加速轮询”逻辑与单测。
  - 存在新增跨平台危险删除命令规则与单测。
- 质量层：
  - 全量门禁无失败（警告可记录但不得阻断判定混淆）。
- 交付层：
  - 触发并完成 `publish-pilot-artifacts`，双平台作业成功。
