## 1. 先回答你最关心的

### 1.1 现在是不是“真实修复”

结论: **部分真实、部分未完成**。

1. 已安装管理里的“卸载”是调用后端真实命令，不是纯前端演示。  
2. 但当前主要是“删配置引用/删 Skill 目录”，**不是完整全局依赖清理**。  
3. `OpenClaw` 有单独“深度卸载”逻辑（进程、npm 包、服务、配置都处理），其他宿主没有同等深度的统一全局清理编排。

### 1.2 现在有没有“全局一键卸载+依赖清理”

结论: **没有完整实现**。

1. 当前前端是“按单个宿主（Host）批量卸载”。  
2. 后端没有“跨全部宿主 + 依赖清理 + 审计报告”的统一命令。  
3. 所以这次要新增的是“全局编排层（Orchestrator）”，不是只改一个按钮文案。

## 2. 用户真实痛点（外部数据）

研究日期: 2026-03-14  
说明: 以下包含官方文档与社区真实投诉。社区投诉用于定位痛点，不等同于厂商官方结论。

### 2.1 高频痛点

1. 误删文件或目录，用户以为是“辅助修改”，实际触发删除动作。  
2. 自动执行删除命令时，用户感知不到风险边界。  
3. 卸载后残留依赖、服务、配置，用户误以为“已经清干净”。  
4. 用户担心“扫到但管不了”，或者“能管但会误删其他软件”。

### 2.2 证据样本

1. Cursor 官方论坛有多起“删除关键文件/项目”的用户报告。  
2. Reddit 上有多起“AI 工具自动删除文件”投诉，集中在未明示审批和回滚困难。  
3. Anthropic 官方 Claude Code 设置文档明确提供敏感文件拒读配置，说明“默认行为边界”必须由产品显式管理。

## 4. 目标定义（这次要做成什么）

在“已安装管理”新增一个真实可执行能力:

1. **一键全局卸载**: 可按“全部 AI 宿主”或“选中的宿主集合”执行。  
2. **对应依赖清理**: 只清理由本工具可归因的 AI 依赖，不碰普通软件。  
3. **已修复不再展示**: 任务完成后默认只显示未解决项。  
4. **全过程可追溯**: 每一步有计划、执行、结果、失败原因和人工兜底建议。

## 5. 边界与防误删规则（AI-only）

### 5.1 允许清理对象

1. MCP 配置项（server 条目）。  
2. Skill 根目录或符号链接。  
3. 能被证据链追溯到目标 AI 组件的依赖项（npm/pip/winget/choco/扩展）。  
4. 对应 AI 宿主服务/计划任务/agent plist。

### 5.2 禁止自动清理对象

1. 与目标 AI 组件无证据关联的全局包。  
2. 系统关键目录、普通应用程序、非 AI 工具。  
3. 无法归因来源的“同名可疑路径”。

### 5.3 必须具备的安全门

1. 先 `dry-run` 计划预览，再执行。  
2. 必须有 runtime approval ticket。  
3. 先写备份清单（至少保存配置变更前镜像）。  
4. 每步失败可继续，但要在报告里标红并给手动命令。

## 7. 执行顺序（固定流程）

1. **Preflight**: 读取最新 discovery + installed 列表，构建 AI Asset Graph。  
2. **Backup**: 备份将修改的配置文件（json/yaml/toml）到 `.agentshield/backups/{run_id}`。  
3. **Stop**: 终止对应 AI 进程和服务（仅目标集合）。  
4. **Detach**: 从配置中移除 MCP/Skill 引用（复用现有 `remove_server_from_config_path`）。  
5. **Uninstall deps**: 走适配器执行依赖清理。  
6. **Remove roots**: 删除 skill root、已确认的宿主缓存或残留目录。  
7. **Verify**: 复扫确认是否仍可解析到目标组件。  
8. **Finalize**: 写报告并刷新前端，默认隐藏已修复项。

## 8. 依赖清理适配器（按来源执行）

### 8.1 npm 适配器

触发条件:

1. 组件来源可追溯为 npm 全局包或 npx 规格。  

执行参考:

1. `npm uninstall -g <pkg>`  
2. `npm prune --dry-run` 先预演，再决定是否执行 prune。

安全要点:

1. 只对“已归因包名”执行，不做全局盲删。  

### 8.2 pip 适配器

触发条件:

1. 组件来源明确来自 pip 包名。  

执行参考:

1. `python -m pip uninstall <package>` 或 `-r requirements`。  

安全要点:

1. 默认先交互确认；仅在明确授权时用 `-y`。  

### 8.3 VS Code / Cursor 扩展适配器

触发条件:

1. 组件对应扩展 ID（`publisher.extension`）可确定。  

执行参考:

1. `code --uninstall-extension <extension-id>`  
2. `code --list-extensions` 做前后对比验证。

### 8.4 winget / Chocolatey 适配器（Windows）

触发条件:

1. 安装来源在 ledger 中可追溯为 winget/choco。  

执行参考:

1. `winget uninstall ...`（必要时加 `--exact`、`--source winget`）。  
2. `choco uninstall ...`，并根据包脚本能力判断是否能真正卸载软件。  

安全要点:

1. choco 可能只移除包记录，不一定卸载系统软件，必须做二次验证并回写报告。

### 8.5 服务适配器

1. macOS: `launchctl` agent/daemon stop/unload。  
2. Linux: `systemctl --user stop/disable`。  
3. Windows: `schtasks /Delete`、`sc delete`（仅命中目标标签）。

### 9.3 文案要求（讲人话）

1. 不写“已执行 X adapter”，写“已清理这个工具的插件配置和相关依赖”。  
2. 失败信息给“下一步手动命令”，不只给错误码。  
3. 英文界面同逻辑，避免中英混杂。

3. 一键执行后到底清了什么、还剩什么、下一步怎么处理。
