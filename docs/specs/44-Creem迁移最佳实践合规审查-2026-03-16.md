## 2. 实现审查：是否符合官方最佳实践

总体结论：**大体符合官方实践，且关键路径正确。**

符合项：

1. Webhook 验签头与算法
   - 代码使用 `creem-signature` + `HMAC-SHA256` 验签，符合官方示例。
   - 代码：`scripts/license-gateway.mjs`
2. 事件映射
   - `checkout.completed` 用于发码，`refund.created` 用于撤销，符合一体化 one-time 模型。
3. Checkout link metadata
   - 参数使用 `metadata[key]` 形式，符合 Creem Checkout Link 文档“Use bracket notation”说明。
4. 生产建议对齐
   - 保持 Webhook 为主，未把前端 `success_url` 跳转当作唯一支付成功依据，符合官方建议。

## 3. 发现的问题（按严重度）

### [P1] 未对“产品计费类型”做强约束，可能导致续费场景漏发码

- 文件：`scripts/license-gateway.mjs:336`
- 问题：目前只处理 `checkout.completed` 发码；若后台把 monthly/yearly 配成 recurring，续费主要走 `subscription.paid`，当前不会自动发续期授权。
- 风险：运营误配置时，用户续费后权益不同步。
- 建议：
  1. 第一阶段强约束 monthly/yearly/lifetime 都必须是 one-time（并在文档/门禁中明确）
  2. 或在 gateway 增加 `subscription.paid` 的续期处理分支

### [P2] metadata 依赖前端 URL 注入，若改用 checkout session URL 可能丢失 sku 语义

- 文件：`src/components/pages/upgrade-pro.tsx:80`
- 问题：当前依赖在 URL 上追加 `metadata[sku_code]`。若以后换成 API 动态创建 checkout 且不传 metadata，则会退化到用 product 名推断计费周期。
- 风险：SKU 命名不规范时，可能错误识别月/年/终身。
- 建议：
  1. 在 gateway 对 `sku_code` 缺失直接报警
  2. 增加 `product_id -> billing_cycle` 显式映射表，避免字符串猜测
